# cf-base — Hetzner baseline server template

A reusable, version-controlled cloud-init configuration + provisioning scripts for Hetzner hosts. Produces a hardened, tailnet-joined Ubuntu 24.04 baseline from scratch every time, with zero snapshot artifact — the YAML and shell scripts in this repo are the source of truth.

## What this is

cf-base captures OS hardening, tailnet membership, and baseline tooling for any Hetzner host. It is applied fresh at provision time via cloud-init — no pre-baked image, no out-of-band manual setup. Once provisioned, a cf-base host has SSH access over Tailscale, UFW firewall enforcement, fail2ban, unattended security updates, Docker, and essential developer tooling (git, uv, Claude Code, jq, ripgrep, fd, and more).

The baseline is specialization-agnostic: it contains zero service code, only the foundation. Downstream services layer on top of a freshly-provisioned cf-base host.

## Prerequisites

Before you start, you need:

- **`hcloud` CLI**, authenticated against your Hetzner account. <https://github.com/hetznercloud/cli>
- **A Tailscale account** with admin access to the ACL. The ACL must declare `tagOwners` for `tag:server` (one-line addition; see `docs/runbook-provision.md` § "Tailscale ACL").
- **An SSH keypair** (ed25519 or RSA). The public key half is what you'll authorize on cf-base hosts.
- **Python 3 with PyYAML** for `cloud-init/build.sh` and `cloud-init/merge.py`. Install with `pip install pyyaml` or `uv pip install pyyaml`.

## Quickstart

1. **Clone and customize.** `cloud-init/base.yaml` is generated locally — your SSH public key never lives in this repo's git history.
   ```bash
   git clone git@github.com:scott-clark-foundry/hetzner-cf-base.git
   cd hetzner-cf-base
   cp cloud-init/operator-pubkeys.txt.example cloud-init/operator-pubkeys.txt
   $EDITOR cloud-init/operator-pubkeys.txt   # paste your SSH public key(s), one per line
   ./cloud-init/build.sh                     # generates cloud-init/base.yaml
   ```
   Re-run `build.sh` after any edit to `base.yaml.in`, `firstboot.sh`, `post-install.sh`, or `operator-pubkeys.txt`.

2. **Generate a Tailscale auth key.** Reusable, tagged (`tag:server`), pre-authorized, non-ephemeral. See `docs/runbook-provision.md` § "Step 1" for the exact admin-console settings and the ACL pre-flight.

3. **Create a per-host overlay.** Copy `user-data/template.yaml` to `user-data/<hostname>.yaml` and fill in the hostname and auth key. (`user-data/cf-*.yaml` is gitignored, so the auth key never enters git history.)

4. **Merge base + overlay into a single provision file.** Use `merge.py` — *not* `cat`; see "Operator notes" below.
   ```bash
   python3 cloud-init/merge.py user-data/<hostname>.yaml > /tmp/<hostname>.yaml
   ```

5. **Provision the host.** No firewall flag at this step — the cloud firewall is applied after Tailnet SSH is confirmed working (Step 8). Provision-time public SSH is your diagnostic escape hatch if cloud-init fails.
   ```bash
   hcloud server create --image ubuntu-24.04 --location hel1 --type <cpx-size> \
     --name <hostname> --user-data-from-file /tmp/<hostname>.yaml
   ```

6. **Wait 2–5 minutes for cloud-init.** Verify the host appears in the Tailscale admin console.

7. **SSH over tailnet:** `ssh deploy@<hostname>`.

8. **Apply the cloud firewall** (once you've confirmed tailnet SSH works):
   ```bash
   hcloud firewall apply-to-resource cf-base-default --type server --server <hostname>
   ```
   See `docs/runbook-provision.md` § "Hetzner Cloud Firewall" for one-time firewall creation.

9. **Run the post-install script** — this is the step that closes public ingress (UFW deny-all on eth0) and turns on fail2ban + unattended-upgrades:
   ```bash
   ssh deploy@<hostname> 'sudo /opt/cf-base/post-install.sh'
   ```

After Step 9, the only path into the host is the tailnet. See `docs/runbook-provision.md` for the full step-by-step including verification checklists (J01, J02) and hardening details.

## What this is NOT

cf-base is baseline infrastructure only. It does **not** include:

- **Service code** — application services (web servers, data pipelines, etc.) are layered onto a cf-base host by whatever follows.
- **Configuration management tooling** — Ansible, Terraform, or other IaC frameworks. At small scale (≤5 hosts), hcloud CLI + cloud-init suffice.
- **Snapshot artifacts** — cloud-init YAML is the source of truth. Every provision boots a fresh Ubuntu 24.04 image and applies cf-base from scratch.
- **Centralized secrets management** — per-host user-data (gitignored) suffices. Service-specific secrets are defined by whatever layers on top.

## Operator notes

**Docker / UFW interaction:** Docker manipulates iptables independently of UFW. cf-base's UFW policy (deny incoming, allow tailscale0 22/tcp + 60000-61000/udp) is correct, but containers started with `-p 80:80` bypass it. Anyone running public-facing containers on a cf-base host must choose between binding to `127.0.0.1` behind a host reverse proxy, binding to the tailscale IP, or accepting Docker's iptables ownership. See `docs/runbook-provision.md` § "Public port exposure" for the three options.
